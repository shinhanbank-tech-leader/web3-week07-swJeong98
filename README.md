# 1 스마트컨트랙트는 왜 "업그레이드가 어려운 구조"인가  
스마트컨트랙트란, 블록체인 상 배포된 코드를 의미하며 배포 즉시 주소를 가지며 코드는 이에 귀속된다.  
변경을 위해서는 새로운 주소에 새로운 코드(계약) 을 두어야 하며 기존 계약에 연결된 데이터를 새 계약으로 변경 하는 과정에서 막대한 양의  
가스비 등 자원이 소모될 수 있다. 이런 구조적 특성에 의해 업그레이드가 어려운 특성을 가진다.  

# 2 delegateCall 기반 업그레이드 방식은 어떤 원리로 동작하는가?  
# 3 Proxy 패턴(로직/데이터 분리)은 어떤 문제를 해결하기 위해 등장했는가 
앞서 보았듯이 새로운 주소에 계약을 올리는 것은 많은 비용을 수반한다. 그렇다고 한번 블록체인 상 기록된 컨트랙트를 변경하지 않고 운영하는 것은  
운영 관점에서 불가피하다. 이 문제를 해결하기 위한 개념이 Proxy 이다. 데이터 저장과 실제 로직(계약)을 분리하는 방식을 의미한다.  

즉, Proxy 컨트랙트를 하나 만들고 이곳에는 계약에 필요한 실제 데이터를 저정하고, 구현부를 담당하는 컨트랙트를 또 다른 주소에 배포하는 방식이다.  
이에 따라 업그레이드 시에는 새로운 로직을 배포하고, Proxy가 바라보는 주소값만 업데이트하면 되기 때문이다.  
이때 Proxy가 로직 컨트랙트를 호출하는 것을 delegateCall이라 하며, 최종적으로는 자신의 상태값만 변경하게 된다.

# 4 은행 시스템에서의 배포/변경 방식과 무엇이 다른가?
# 5 은행 시스템에서는 변경 시 어떤 통제(승인, 검증, 롤백 등)를 수행하는가 
1) 변경 방식  
은행 시스템은 내부 절차를 통해 서버에 올라간 소스코드 등을 직접 변경하고, 스마트 컨트랙트는 로직 컨트랙트를 배포하고 Proxy가 가리키는 로직 컨트랙트의 주소를 변경한다.  
2) 승인 프로세스  
은행 시스템은 내부 결재 및 배포 파이프라인을 거치고, 스마트 컨트랜트는 정책에 따라 거버넌스 정책(DAO) 등을 통해 승인이 이뤄진다.  

3) 배포 내용 확인  
은행 시스템은 내부 직원만 형상관리, 동료코드리뷰 등 통제에 관여하나 스마트 컨트랙트는 온체인 상에서 누구나 확인 가능하여 통제에 참여할 수 있다.  

# 6 스마트컨트랙트 업그레이드 방식에서 발생할 수 있는 위험은 무엇인가  
1) 저장소 충돌 문제
구현 컨트랙트 간 같은 변수를 다른 용도로 사용할 때 발생할 수 있다. 예를 들어 기존 로직에서는 첫 번째 변수에 주소를 저장했는데,  
업데이트 로직에서 고객잔액을 저장할 경우 주소 데이터가 잔액으로 변경되어 시스템 제어권이 상실될 수 있다.  
2) 함수 선택자 충돌 문제  
프록시와 로직 컨트랙트의 함수 이름이 컴퓨터 해석 단계에서 겹치는 현상을 의미한다. 이더리움의 경우 함수 이름을 4byte의 해시 값으로 구분하는데,  
이 값이 같을 경우 다른 함수를 호출 해 전혀 다른 방향으로 거래가 진행될 위험이 있다.  

# 7 두 구조를 비교했을 때 운영·통제 관점에서 가장 큰 차이는 무엇인가 
가장 큰 차이는 통제의 주체 및 강제성에 있다.  

은행 시스템은 내부 통제 절차에 기반하며 시스템 장애 또는 사고 발생시 물리적인 복구가 가능하며 때로는 제도적 범위 안에서 운영 위험을 직접 해결한다.  
반면 스마트 컨트랙트는 오로지 코드 기반의 신뢰를 구축하므로 실수 또는 해킹 발생 시 블록체인 구조상 완벽한 복구가 불가하다.  

이런 차이로 인해, 운영 시 발생할 수 있는 변수에 대한 유연성 측면에서는 기존 은행 시스템이 더욱 장점이 크다고 생각한다.
